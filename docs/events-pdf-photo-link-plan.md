## 1. 目的

現在の `#/events` は、募集カードごとに活動種別の共通イメージ写真を表示している。これは安全だが、PDF `202605.pdf` の活動レポートと直接つながっている印象は弱い。

次の段階では、PDF内の実活動写真をイベントカードに使い、閲覧者が「この募集は過去にこういう活動として行われている」と具体的に想像できる状態にする。

## 2. 現状

### 既存実装

- `src/data/siteContent.ts`
  - `recruitmentSnapshot.items` に `image` が追加済み。
  - 現在は `eventImages.forest` / `eventImages.field` / `eventImages.regional` の共通写真を割り当てている。
- `src/App.tsx`
  - `RecruitmentCard` がカード上部に `item.image` を表示する。
  - Desktop / Mobile で横スクロールなし、現行の募集10カード全て画像表示済み。

### 現在の問題

- 写真が共通素材なので、個別イベントとの結びつきが弱い。
- PDF内の活動レポート写真とWebサイト上の募集カードがまだ直接対応していない。
- PDF写真には人物が写るものが多く、Web再利用時の許諾・顔写り確認が必要。

## 3. PDF写真の利用方針

### 基本方針

1. PDF写真は「活動イメージ」として使う。
2. 募集中イベントの現地写真として断定しない。
3. 顔が大きく判別できる写真は、許諾確認または顔なし加工後に使う。
4. まずは `#/events` の募集カード写真をPDF由来に置き換える。
5. 写真の出典・用途・安全判断をデータに残す。

### 表示ラベル

現在のカード上部ラベルは、以下のように変更する。

- `過去活動の様子`
- `森林の楽校 活動イメージ`
- `田畑の楽校 活動イメージ`
- `地域交流 活動イメージ`

カード上の推奨は `過去活動の様子`。閲覧者に自然で、現在募集中イベントの現地写真だと誤認されにくい。PDFとの関係は、セクション説明で「PDF掲載の過去活動写真を参考イメージとして使用」と補足する。

## 9. 実装手順

### Step 1: PDF写真の切り出し

- `page_08.png` がない場合は、Section 4 の「PDFページ画像の再生成手順」でPDFからページ画像を再生成する。
- `tools/extract_events_pdf_photos.py` を作成し、Python + Pillow で `tmp/pdfs/202605_pages/page_08.png` から対象写真を切り出す。
- 切り出し座標はスクリプト内の辞書で管理し、写真ID・PDF上の活動名・出力ファイル名を対応させる。
- 画像は `16:9` に整える。
- 顔が大きい写真は避ける、または人物が特定されにくい位置でトリミングする。
- 最終採用が決まるまでは `tmp/events-photo-crops/` に保存し、採用が確定した画像だけ `public/assets/photos/events/` にコピーする。
- 既存画像を上書きする場合は `--force` オプションを必須にする。

### Step 2: 写真確認シート作成

実装前に、切り出した写真の一覧確認用画像を作る。

保存先:

```text
juon-site/tmp/events-pdf-photo-contact-sheet.png
```

`tmp/` はルート `.gitignore` の対象なので、中間確認用の大きなPNGを誤ってコミットしにくい。`output/` を使う場合は、生成前に `output/` または対象ファイルを `.gitignore` に追加する。

確認観点:

- 活動名と写真が合っているか
- 顔が大きく写っていないか
- カードにしたとき主題が見切れないか
- 暗すぎないか、文字ラベルと干渉しないか

確認シートまたは `docs/events-photo-sources.md` には、各画像ごとに以下の判定列を持たせる。

```text
採用 / 不採用 / 要トリミング / 要許諾確認
```

`要許諾確認` の写真は `public/assets/photos/events/` に入れない。`要トリミング` は再切り出し後に再確認する。

### Step 3: 出典・安全判断の永続化

- `docs/events-photo-sources.md` を作成する。
- 最終採用画像だけを記録する。
- 記録項目は、画像ID、採用先イベント、PDFページ、PDF上の活動名、判定、トリミング方針、安全判断、備考とする。
- `tmp/` の確認シートはコミットしない前提のため、判断根拠として扱わない。

### Step 4: `siteContent.ts` の写真差し替え

- `eventImages` を共通写真からPDF由来写真へ変更する。
- または `eventPhotoImages` を新設して、イベントごとに `image` を個別指定する。

推奨は後者。

理由:

- 共通カテゴリ写真とPDF由来写真を分けられる。
- 許諾未確認の写真を一時的に戻しやすい。
- イベントごとの対応関係が読みやすい。

### Step 5: 表示文言の調整

カード上のラベルを `過去活動の様子` にする。

必要なら `statusNote` またはカード下部に次の補足を入れる。

```text
写真はPDF掲載の過去活動写真を参考イメージとして使用しています。募集内容の詳細は公式ページで確認してください。
```

ただし全カードに補足を増やすと重くなるため、まずはセクション上部の説明文に1回だけ置く方がよい。

### Step 6: 表示確認

必須確認:

- `npm run build`
- 開発サーバー起動: `npm run dev -- --host 127.0.0.1`
- `http://127.0.0.1:5173/#/events`
- Desktop `1440x900`
- Mobile `390x844`
- 全カード画像読込
- 横スクロールなし
- 画像ラベルが写真の主題を隠していない
- `alt` が活動内容を説明している

## 10. リスクと判断基準

### リスク1: 肖像権

PDF掲載済みでも、Webサイト上で再利用できるとは限らない。顔が大きく写る写真は使わない。集合写真や講座写真は特に注意する。

判断:

- 顔が主題になっている写真: 使わない
- 後ろ姿・遠景・手元作業中心: 採用可
- オンライン画面・個人名が見える写真: 使わない

### リスク2: 募集内容との誤解

PDF写真は過去活動の写真であり、現在募集中のイベントそのものではない場合がある。

判断:

- 名称一致: 過去の同名活動イメージとして使う
- 名称不一致だがカテゴリ一致: 活動イメージとして使う
- 地域も活動種別も異なる: 使わない

### リスク3: 画質

PDFページからの切り出しのため、元写真より解像度が落ちる可能性がある。

判断:

- カード表示で粗さが目立たない: 採用
- ヒーローや大判表示で粗い: 不採用
- まずはイベントカード用途に限定する

## 12. Go / No-Go

Go条件:

- 切り出し後の確認シートで、最低6枚が活動一致・画質・顔写りの確認を通過する
- `docs/events-photo-sources.md` に、公開採用する全画像の出典・用途・安全判断が記録されている
- 顔・個人情報リスクが高い写真を避けられる、または安全なトリミングに変更できる
- 確認シートなどの中間成果物が `tmp/` 配下、または `.gitignore` 対象である
- イベントカードで画像が自然に表示される
- `npm run build` が成功する

No-Go条件:

- 採用候補の多くが顔中心で、トリミングしても人物が特定される
- PDF切り出し画像が粗く、カードでも品質が落ちる
- 写真と募集内容の対応が曖昧で誤解を生む

## 13. 推奨結論

実装は Conditional Go。進めてよいが、写真は「現在募集中イベントの現地写真」ではなく「過去活動の様子」として扱う。実装前に、PDFページ画像の再生成手順、採用判定、永続的な出典・安全判断の記録をそろえる。

最初に使うべき写真は、名称一致している以下4件。

1. 風の谷 森林の楽校
2. 北信りんごの里 田畑の楽校
3. 安曇野ツアー
4. 農業体験＆竹林整備 @ 飯島農園

残りの森林系イベントは、神の泉・四万十川・森林ボランティア入門講座などの写真を「活動イメージ」として割り当てる。同一写真の重複は、名称一致写真がない場合に限り許容するが、隣接カードで同じ写真が並ばないように最終表示順で確認する。
