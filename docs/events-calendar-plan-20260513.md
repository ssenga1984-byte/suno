# `#/events` カレンダー追加 実装計画書

作成日: 2026-05-13  
対象ページ: `http://127.0.0.1:5173/#/events`  
対象期間: 2026年5月、2026年6月  
対象データ: `src/data/siteContent.ts` の `recruitmentSnapshot.items`

## 1. 目的

イベントページ上部に、今月と来月の開催日が一目でわかるカレンダーを追加する。  
現在の `#/events` は「日程から見つける」導入、地域リンク、直近募集、月別リストを持っているが、日付単位でどの日に開催があるかを視覚的に把握しにくい。ページ上部に2か月分のカレンダーを置き、開催日のセルにイベント件数と短いタイトルを表示して、下部の募集カードへ自然に進める導線を作る。

## 2. 現状整理

- `#/events` は `src/App.tsx` の `EventsPage` で描画されている。
- イベントデータは `recruitmentSnapshot.items` に集約されている。
- 日付キーとして `startDate`、任意の終了日として `endDate` がある。
- 既存の月別表示は `groupRecruitmentByMonth()` で `monthLabel` ごとにまとめている。
- 地域絞り込みは `#/events?region=...` で `visibleItems` を切り替える実装がある。
- `#/events` は主要DOMルートなので、PNGモック依存ではなくReact/Tailwindで拡張する。

## 3. 仕様方針

### 表示対象

- 基準日は `recruitmentSnapshot.checkedAt` を使う。
- 今回のスナップショットでは `2026-05-12` が基準なので、表示月は `2026-05` と `2026-06` に固定される。
- 7月以降のイベントは既存の月別リストには残すが、上部カレンダーには出さない。
- 地域絞り込み中は、カレンダーも `visibleItems` に連動して該当地域のイベントだけを表示する。

### イベント日の扱い

- `startDate` の日付セルにイベントを表示する。
- `endDate` がある複数日イベントは、初期実装では開始日のセルに表示し、セル内に「2日間」などの補足を出す。
- 期間中の全日セルを塗る実装は次フェーズ扱いにする。初期実装でここまで行うと、狭いモバイル幅で情報量が過多になりやすい。

### UI

- `PageBanner` 直下、既存の「日程から見つける」セクション内またはその直後に配置する。
- 2か月をデスクトップでは2カラム、モバイルでは1カラム縦積みにする。
- 曜日見出しは7列固定にし、各日セルのサイズがイベント有無で大きく崩れないようにする。
- 開催日のセルは `forest.leaf` 系の左線または薄い背景で強調し、CTA色の `forest.gold` は件数バッジ程度に限定する。
- 各開催日セルには以下を表示する。
  - 日付
  - イベント件数
  - 最大2件までの短いタイトル
  - 3件以上は「ほかN件」
- セル内のタイトルはリンクにせず、クリック領域の複雑化を避ける。詳細確認は直下の既存カード一覧へ誘導する。

## 5. 実装手順

1. `EventsPage` の `visibleItems` 算出後に、2か月分のカレンダー表示用データを生成する。
2. `PageBanner` の直後、既存導入セクションの上部に `EventsCalendarSection` を追加する。
3. `EventsCalendarSection` で `2026年5月`、`2026年6月` の2つの月カレンダーを描画する。
4. 各月は日曜始まりの7列グリッドにする。
5. `startDate` が該当するイベントを日付セルへまとめる。
6. `endDate` があるイベントはタイトル横または補足行に期間表示を付ける。
7. 地域フィルタ中の表示として、カレンダー上部に「選択地域のイベントだけを表示中」とわかる短い補足を出す。
8. 空の月または該当イベントがない地域フィルタでは、カレンダー自体は残し、該当日なしの状態を明示する。

## 6. デザイン制約

- 既存のDOM実装方針を維持し、PNGモックを新規正本にしない。
- カード内カードにはしない。カレンダー月ごとは独立したパネルとして扱う。
- 角丸を増やさず、既存の直線的なJUONサイトの調子に合わせる。
- 装飾目的のグラデーションや抽象背景は追加しない。
- モバイルで7列が窮屈になるため、セル内タイトルは1件目中心に抑え、文字がはみ出す場合は短く折り返す。

## 7. アクセシビリティ

- カレンダー全体に `aria-labelledby` を設定する。
- 月ごとの見出しを `h3` にする。
- 開催日のセルには視覚表示だけでなく、スクリーンリーダー向けに「5月23日、2件のイベント」のような説明を含める。
- イベントタイトルは既存カードにリンクがあるため、カレンダー内ではリンクを増やしすぎない。
- `document.title` は既存の `#/events` 設定を維持する。

  - `aria-label` またはスクリーンリーダー用説明がある

## 9. リスクと対策

### リスク1: 「今月・来月」の基準が曖昧

対策: 実装では `recruitmentSnapshot.checkedAt` を基準にする。現在の確認日が `2026-05-12` なので、5月・6月を表示する。将来スナップショットを更新したら自動で対象月も変わる。

### リスク2: 複数日イベントの見せ方が誤解を生む

対策: 初期実装では開始日のみに表示し、「2日間」などの補足を付ける。全日塗りつぶしは次フェーズに分ける。

### リスク3: モバイルで情報過多になる

対策: セル内表示は最大2件までに制限し、詳細は既存カード一覧へ誘導する。長いタイトルは折り返しと行数制限を前提にする。

### リスク4: 地域フィルタとの不整合

対策: `visibleItems` を唯一の入力にして、カード一覧とカレンダーが同じフィルタ結果を使うようにする。

## 10. Go / No-Go

Go条件:

- カレンダーが `recruitmentSnapshot.items` から生成され、手書きの重複データを持たない。
- 2026年5月・6月だけが上部カレンダーに表示される。
- 地域フィルタ時にもカレンダーが同じ絞り込みに連動する。
- Desktop/Mobileで横スクロールやセル内テキストの重なりがない。
- `npm run build` が成功する。

No-Go条件:

- カレンダー専用にイベント情報を二重管理する。
- 7月以降のイベントまで上部カレンダーに混ざる。
- モバイルで7列グリッドが横にはみ出す。
- 既存のイベントカード一覧、地域フィルタ、外部リンク導線を壊す。

## 11. 推奨結論

実装はGo。  
既存の `EventsPage` はすでに `visibleItems`、`upcomingItems`、`monthlyGroups` を持っており、カレンダー追加に必要な日付データも `RecruitmentItem` に揃っている。新規データ取得やページ構造の大規模変更は不要で、`EventsPage` 上部に表示専用コンポーネントを追加するのが最小変更になる。
